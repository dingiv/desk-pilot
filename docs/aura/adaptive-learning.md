# 自适应学习：存疑提问 + 纠错闭环 + 动态微调

记录两环想法（2026-07-14 讨论）：Stage2 存疑检测 → 秘书主动提问 → 用户纠错 → 沉淀为用户专属能力（热词 / LoRA），随使用越来越准。这是"私人秘书 vs 通用转写器"的核心差异化护城河。

---

## 一、问题：语音识别的"音→字歧义"是 ASR 层面无解的

ASR（SenseVoice 等）是"听音记字"。人说话时，**音到字的映射有歧义，真正意思要靠上下文 + 世界知识**。

典型案例（贪吃蛇录音）：
- 录音发音确实是 "B 位引擎"，ASR 正确地记成了"B位"
- 但用户指的是 Rust 生态著名的 **Bevy 引擎**（发音近似"B位"）
- SenseVoice 没错——它不知道用户在聊 Rust 游戏开发、不知道有 Bevy

同类：`rost→rust`、`蛇声→蛇身`、`计分起→计分器`、`readdme→README`、人名、项目名、术语。**这些问题 ASR 永远解不了**，因为缺的是"世界知识 + 用户语境"。

---

## 二、想法 A：Stage2 存疑检测 → 秘书主动提问（不瞎猜）

把"无解的硬猜"转化为"可解的产品能力"——AI 秘书不瞎猜、不沉默，而是**主动澄清**。

### 流程
```
Stage1 原文（含误读 "B位"）
   ▼
Stage2 整流 + 存疑检测：
   - calibrated_text: "...采用 2D 引擎..."（保留原音写法，不硬改）
   - uncertainties: [{span:"B位", context:"Rust 游戏引擎", note:"疑似引擎名，写法存疑"}]
   ▼
秘书把存疑项变成自然提问：
   "你说的'采用 B 位引擎'——是指 Rust 的 Bevy 引擎吗？"
   ▼
用户确认/纠正 → 得到正确答案 (B位 → Bevy)
```

### Stage2 JSON 扩展（uncertainties 字段）
```json
{
  "calibrated_text": "...图形渲染采用 2D 引擎...",
  "intent": "task",
  "reply": "...",
  "task": {...},
  "uncertainties": [
    {"span": "B位", "guess": "Bevy", "confidence": "low",
     "why": "Rust 游戏开发语境，疑似引擎名，发音不确定"}
  ]
}
```
提示词要求 Qwen：**整流时遇到可能是专名/术语但写法存疑的词，不硬改，标进 uncertainties**（原音写法留在 calibrated_text，附上猜测）。

### 关键现实：1.7B 小模型世界知识有限
- Qwen3-1.7B **能标出存疑点**（"这里可能是专名"）——这件事做得到
- 但它的 `guess` 可能不准（不一定知道 Bevy）——存疑检测 ≠ 猜对答案
- 真正"猜对 Bevy"要靠 Stage3 / 更大模型 / 检索（见下）

### 这一步的现状
- ✅ 现在就能做（改提示词 + Decision 结构加 uncertainties）
- 用 hungry_snake 转写离线测，验证 Qwen 标不标得出"B位"存疑

## 三、想法 B：纠错闭环 → 动态微调（越用越准）

存疑提问是起点。真正的价值是**把每次纠错沉淀成模型的专属能力**。

### 完整闭环
```
日常对话 → ASR 听错(B位) → Stage2 标存疑 → 秘书提问 → 用户确认(Bevy)
                                                          │
                     ┌────────────────────────────────────┘
                     ▼
          纠错样本: (音频"bèi wèi" → "Bevy", 上下文="Rust 游戏开发")
                     │  攒一批（后台、定期）
                     ▼
          LoRA 微调 ASR（或热词偏置）→ 部署 → 下次直接出"Bevy"
```
每一轮都让模型更懂**这个用户**的工作语境（专有名词、口癖、口音、习惯缩写）。

---

## 四、三个关键技术现实（决定可行性）

### 现实 1：SenseVoice 不适合直接 LoRA
- SenseVoice 是 **ONNX 推理模型**，原训练在 PyTorch/FunASR
- LoRA 微调要回 PyTorch + FunASR 训练框架，**不能在 ONNX/ort 层面做**
- SenseVoice 多任务架构（ASR+语种+情感+事件）微调复杂、LoRA 支持不成熟
- → 要微调，换 **Whisper / Paraformer**（有成熟 PEFT/LoRA pipeline）

### 现实 2：更轻的替代——专属热词表（biasing / contextual biasing）
动 LoRA 之前，有立竿见影、零训练成本的方案，覆盖 80% 专有名词问题：
- 每次纠错(B位→Bevy)写入用户热词表
- ASR 推理时把热词表作**上下文偏置**喂模型（sherpa-onnx 的 keyterms/keyword biasing）
- 不用训练、即时生效、可增删
- sherpa-rs 的 SenseVoice/Zipformer `STTCapabilities.keyterms` 待验证（**关键前置探查**）

**LoRA 留给"口癖/口音/句式"这类热词解决不了的深层模式。**

### 现实 3：LoRA 微调是"重"操作，要控节奏
- **数据要够**：单条不够，攒几十~几百条同类才值得微调（否则过拟合）
- **训练在哪**：RTX 5070 Ti 本地可微调小模型（Whisper-tiny/base LoRA 几分钟/条）；复用 OmniParser venv 的 torch cu128
- **离线后台**：不阻塞对话，定期批量训，热替换权重
- **回滚/版本**：微调可能变差（灾难性遗忘），要 A/B + 保 baseline

## 五、演进路线（三层，由轻到重）

### 第 1 层（现在能做）：热词偏置
- ASR 接 keyterms，喂用户热词表
- 纠错时即时写入热词表
- **覆盖**：专有名词、术语、人名（Bevy、项目名、同事名）
- **前置**：验证 sherpa-rs SenseVoice 的 keyterms 支持

### 第 2 层（下一步）：纠错样本库 + 检索增强
- 每次纠错存 `(音频片段, 正确文本, 上下文)` 三元组
- 推理时检索相似上下文样本，作 few-shot 提示给 Stage2 校准
- **覆盖**：同音字歧义的上下文化解（不训练，用检索）
- 不需训练，用向量检索（用户专属 RAG）

### 第 3 层（远期，LoRA 愿景）：定期后台 LoRA
- 攒够样本后，后台微调（Whisper/Paraformer 的 LoRA，**非 SenseVoice**）
- 本地训练（5070Ti）→ 热替换 → A/B
- **覆盖**：口癖、口音、句式、个人语言习惯

---

## 六、共同基础：纠错样本采集与存储

三层都需要同一个数据源：**纠错样本**。而纠错样本来自"存疑→提问→用户确认"。所以两件事的交汇点：

**先打通"存疑检测 + 提问 + 纠错采集"**——这是第 1 层和第 3 层的共同基础。采集的 `(音频, 原识别, 正确文本, 上下文)` 进专门 store：
- 第 1 层用它生成热词
- 第 3 层用它训 LoRA

### 落地下一步（建议顺序）
1. **Stage2 加 uncertainties 输出**（存疑检测）—— 立刻可用
2. **`corrections` 表**（SQLite：chunk_id / 音频片段引用 / 原文本 / 正确文本 / 上下文 / 时间戳）
3. **ASR 接 keyterms** —— 验证热词偏置在 SenseVoice/sherpa 能否用（决定第 1 层可行性，**最该先探查**）

---

## 七、竞争力判断

现有语音助手（讯飞/搜狗/Whisper）面对音同字歧义只有两条路：硬猜（常错）或照发音记（B位）。**没有一个会主动发问**，更没有"越用越懂你"的个性化。

本设计让秘书从"被动转写器"升级成"主动澄清 + 自我进化的协作者"——正是 chat.txt "长期秘书 vs 金鱼伙伴"分水岭的体现，是核心护城河。
